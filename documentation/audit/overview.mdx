<Note>
Audit is a frozen secondary library. It is not mounted as `ctpf audit`, is not the CTPF research
contribution, and should not expand in competition with dedicated external scanners.
</Note>

The retained library connects to an MCP server, enumerates tools, resources, and prompts, runs
registered static or active checks, and exports findings in structured formats. These observations
do not become controlled experiment results or CTPF conclusions automatically.

## Minimal library use

```python
import asyncio

from ctpf.audit.orchestrator import run_scan
from ctpf.audit.reporting.json_report import generate_json_report
from ctpf.mcp.connection import MCPConnection


async def main() -> None:
    async with MCPConnection.stdio("python", ["my_server.py"]) as connection:
        result = await run_scan(connection)
    generate_json_report(result, "results/scan.json")


asyncio.run(main())
```

## Pipeline

1. **Connect** through stdio, SSE, or Streamable HTTP.
2. **Enumerate** the server's tools, resources, and prompts.
3. **Scan** with all registered checks or a selected `check_names` subset.
4. **Report** as JSON, SARIF, HTML, NDJSON, or CSV.

Active scanners invoke target tools and may cause side effects. Use them only against authorized,
controlled systems.

## Scanner categories

The library retains ten scanner registrations corresponding to its OWASP MCP Top 10 mapping:
token exposure, permissions, tool poisoning, supply-chain metadata, injection, prompt-injection
patterns, authentication, audit/telemetry, shadow-server indicators, and context sharing.

These are scanner observations and taxonomy mappings. For example, detecting prompt-like text in a
tool result does not establish that an agent consumed it or produced an external effect.

## Reports and persistence

Report generators consume `ScanResult` and write explicit output files. `persist_scan()` can map
scan findings into the operational SQLite schema, but persistence is optional and remains separate
from `ctpf experiment` evidence directories.
