---
title: MCP Transport Types
description: Configure how CTPF connects to MCP servers
---

CTPF's MCP and proxy layers support stdio, legacy SSE, and Streamable HTTP.

| Transport | Flag value | Connection | Typical use |
|-----------|------------|------------|-------------|
| **stdio** | `stdio` | Spawn a controlled child server over stdin/stdout | Local MCP servers and packaged experiment fixtures |
| **SSE** | `sse` | Legacy HTTP Server-Sent Events | Servers that still expose the older MCP transport |
| **Streamable HTTP** | `streamable-http` | Current bidirectional HTTP transport | Modern MCP servers and the loopback experiment seam |

## stdio

```bash
ctpf proxy start --transport stdio --target-command "python my_server.py"
```

The proxy owns the child-process lifecycle and relays MCP messages between the controlled client
and server. The cascade experiment uses this transport upstream to its honest packaged fixture.

## SSE

```bash
ctpf proxy start --transport sse --target-url "https://mcp-server.example.com/sse"
```

Use SSE only for a server that implements the legacy transport.

## Streamable HTTP

```bash
ctpf proxy start --transport streamable-http --target-url "https://mcp-server.example.com/mcp"
```

The cascade experiment exposes its client-facing proxy endpoint over Streamable HTTP on
`127.0.0.1`. Both the driven-inference and Claude Code targets connect through that observable,
mutable loopback seam.

CTPF-owned local HTTP listeners always bind to `127.0.0.1`; they never bind to an external
interface.
