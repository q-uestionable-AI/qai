# MCP Transports Configuration

Configure how qai communicates with Model Context Protocol (MCP) servers for audit, proxy, and other module integrations.

## Overview

qai supports multiple MCP transport types for different deployment scenarios:

- **Stdio**: Direct process-to-process communication (default, local)
- **SSE**: Server-Sent Events over HTTP (streaming, firewall-friendly)
- **Streamable HTTP**: Bidirectional HTTP with streaming (cloud-native)

Each transport has specific use cases, performance characteristics, and configuration requirements.

## Transport Types

### Stdio Transport

**Best for**: Local development, single-machine setups, direct process integration

Stdio transports communicate directly between processes via standard input/output.

```json
{
  "transport": {
    "type": "stdio",
    "command": "qai-audit-mcp",
    "args": ["--config", "/etc/qai/audit.json"]
  }
}
```

**Advantages**:
- No network overhead
- Lowest latency
- Simplest setup
- Direct process control

**Disadvantages**:
- Can't cross machine boundaries
- Single connection per process
- No network resilience

**When to use**:
- Development and testing
- Single-machine deployments
- Direct application integration
- CLI tools and scripts

### SSE Transport

**Best for**: Remote connections, firewalls, read-heavy workloads, GitHub Actions

Server-Sent Events allow HTTP connections with server-initiated messages.

```json
{
  "transport": {
    "type": "sse",
    "url": "https://mcp-server.example.com/events",
    "method": "POST",
    "auth": {
      "type": "bearer",
      "token": "sk-sse-..."
    },
    "timeout_ms": 30000
  }
}
```

**Advantages**:
- Works through HTTP proxies and firewalls
- Stateless (each request independent)
- Simple authentication (HTTP Bearer tokens)
- Built-in keep-alive

**Disadvantages**:
- Slightly higher latency than stdio
- Unidirectional communication (mostly server→client)
- Less efficient for bidirectional streaming

**When to use**:
- Cloud deployments
- CI/CD pipelines (GitHub Actions, GitLab CI)
- Behind HTTP proxies
- Firewall-restricted networks
- Audit module connections

### Streamable HTTP Transport

**Best for**: High-throughput bidirectional communication, enterprise deployments

HTTP with streaming support allows true bidirectional communication over HTTP.

```json
{
  "transport": {
    "type": "streamable_http",
    "url": "https://mcp-server.example.com/stream",
    "method": "POST",
    "auth": {
      "type": "oauth2",
      "client_id": "qai-client",
      "client_secret": "secret"
    },
    "timeout_ms": 60000,
    "buffer_size": 65536,
    "keepalive_interval_ms": 30000
  }
}
```

**Advantages**:
- True bidirectional streaming
- Efficient for large payloads
- HTTP/2 support
- Advanced authentication options

**Disadvantages**:
- More complex setup
- Higher memory overhead
- Requires stream-aware server

**When to use**:
- Enterprise deployments
- High-throughput scenarios
- Proxy module with large datasets
- Long-lived connections

## Configuration

### Configuration File Format

Configure transports in `qai.json` or `config.yaml`:

```json
{
  "modules": {
    "audit": {
      "transport": {
        "type": "sse",
        "url": "https://audit-mcp.example.com/events",
        "auth": {
          "type": "bearer",
          "token": "${AUDIT_MCP_TOKEN}"
        }
      }
    },
    "proxy": {
      "transport": {
        "type": "streamable_http",
        "url": "https://proxy-mcp.example.com/stream",
        "auth": {
          "type": "bearer",
          "token": "${PROXY_MCP_TOKEN}"
        }
      }
    }
  }
}
```

<Note>
Configuration values with `${VARIABLE}` syntax are replaced with environment variables at runtime.
</Note>

### Environment Variables

Configure transports using environment variables:

```bash
# Stdio (default)
export QAI_AUDIT_TRANSPORT=stdio

# SSE
export QAI_AUDIT_TRANSPORT=sse
export QAI_AUDIT_MCP_URL=https://audit-mcp.example.com/events
export QAI_AUDIT_MCP_TOKEN=sk-sse-...

# Streamable HTTP
export QAI_AUDIT_TRANSPORT=streamable_http
export QAI_AUDIT_MCP_URL=https://audit-mcp.example.com/stream
export QAI_AUDIT_MCP_AUTH_TYPE=oauth2
export QAI_AUDIT_MCP_CLIENT_ID=...
export QAI_AUDIT_MCP_CLIENT_SECRET=...
```

### Command-Line Arguments

Override transports via CLI:

```bash
# Use SSE transport
qai audit --transport sse \
  --mcp-url https://audit-mcp.example.com/events \
  --mcp-token sk-sse-...

# Use Streamable HTTP with OAuth2
qai audit --transport streamable_http \
  --mcp-url https://audit-mcp.example.com/stream \
  --mcp-auth oauth2 \
  --mcp-client-id qai-client \
  --mcp-client-secret secret
```

## Authentication

### Bearer Token Authentication

Simple token-based authentication for SSE:

```json
{
  "auth": {
    "type": "bearer",
    "token": "sk-sse-your-token-here"
  }
}
```

```bash
export QAI_AUDIT_MCP_TOKEN=sk-sse-your-token-here
```

### OAuth2 Authentication

For enterprise deployments requiring OAuth2:

```json
{
  "auth": {
    "type": "oauth2",
    "client_id": "qai-app",
    "client_secret": "secret-key",
    "token_url": "https://auth.example.com/token",
    "scopes": ["audit:read", "audit:write"]
  }
}
```

```bash
export QAI_AUDIT_MCP_CLIENT_ID=qai-app
export QAI_AUDIT_MCP_CLIENT_SECRET=secret-key
export QAI_AUDIT_MCP_TOKEN_URL=https://auth.example.com/token
```

### mTLS Certificates

For secure connections with mutual TLS:

```json
{
  "auth": {
    "type": "mtls",
    "cert_file": "/etc/qai/client.crt",
    "key_file": "/etc/qai/client.key",
    "ca_file": "/etc/qai/ca.crt"
  }
}
```

```bash
export QAI_AUDIT_MCP_CERT_FILE=/etc/qai/client.crt
export QAI_AUDIT_MCP_KEY_FILE=/etc/qai/client.key
export QAI_AUDIT_MCP_CA_FILE=/etc/qai/ca.crt
```

## Connection Examples

### Local Development (Stdio)

```bash
# Run audit using stdio (direct process)
qai audit --framework owasp-top-10 --transport stdio
```

Configuration file:
```json
{
  "modules": {
    "audit": {
      "transport": {
        "type": "stdio",
        "command": "qai-audit-server"
      }
    }
  }
}
```

### GitHub Actions (SSE)

```yaml
- name: Run qai audit
  run: |
    qai audit \
      --transport sse \
      --mcp-url ${{ secrets.AUDIT_MCP_URL }} \
      --mcp-token ${{ secrets.AUDIT_MCP_TOKEN }}
  env:
    QAI_AUDIT_MCP_URL: https://audit-mcp.example.com/events
    QAI_AUDIT_MCP_TOKEN: ${{ secrets.AUDIT_MCP_TOKEN }}
```

### Enterprise Kubernetes (Streamable HTTP + OAuth2)

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: qai-config
data:
  qai.json: |
    {
      "modules": {
        "audit": {
          "transport": {
            "type": "streamable_http",
            "url": "https://audit-mcp:8443/stream",
            "auth": {
              "type": "oauth2",
              "client_id": "qai-k8s",
              "token_url": "https://oidc.example.com/token"
            }
          }
        }
      }
    }
```

Deployment:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: qai-audit
spec:
  template:
    spec:
      containers:
      - name: qai
        env:
        - name: QAI_AUDIT_MCP_CLIENT_SECRET
          valueFrom:
            secretKeyRef:
              name: qai-secrets
              key: mcp-secret
```

## Proxy Module Transports

The proxy module uses its own transport for intercepted traffic:

```json
{
  "modules": {
    "proxy": {
      "transport": {
        "type": "streamable_http",
        "url": "https://proxy-mcp.example.com/stream",
        "auth": {
          "type": "bearer",
          "token": "${PROXY_MCP_TOKEN}"
        },
        "buffer_size": 131072,
        "keepalive_interval_ms": 30000
      }
    }
  }
}
```

### Proxy Transport Selection

- **Development**: Stdio (direct connection to proxy server)
- **Cloud**: SSE (stateless, scalable)
- **High-volume traffic**: Streamable HTTP (bidirectional streaming)

## Performance Tuning

### Timeout Configuration

Adjust timeouts based on network conditions:

```json
{
  "transport": {
    "type": "sse",
    "url": "https://mcp-server.example.com/events",
    "timeout_ms": 30000,
    "connect_timeout_ms": 5000,
    "idle_timeout_ms": 60000
  }
}
```

### Buffer Sizing

Optimize for throughput vs. memory:

```json
{
  "transport": {
    "type": "streamable_http",
    "url": "https://mcp-server.example.com/stream",
    "buffer_size": 65536,
    "max_buffer_pools": 10
  }
}
```

### Connection Pooling

Reuse connections for multiple requests:

```json
{
  "transport": {
    "type": "streamable_http",
    "url": "https://mcp-server.example.com/stream",
    "max_connections": 10,
    "keepalive_interval_ms": 30000,
    "idle_connection_timeout_ms": 300000
  }
}
```

## Health Checking

### Transport Health Status

Check transport connectivity:

```bash
# Test transport
qai config test-transport audit --verbose

# Output:
# ✓ Connecting to MCP server...
# ✓ Authentication: Valid
# ✓ Round-trip latency: 145ms
# ✓ Transport healthy - ready to use
```

### Monitoring Metrics

Monitor transport performance:

```bash
# View transport metrics
qai status --show-metrics

# Output shows:
# - Connection uptime
# - Message throughput
# - Latency (min/avg/max)
# - Error rates
# - Authentication refreshes
```

## Troubleshooting

### Connection Failed

<Warning>
If connection fails, verify:
1. MCP server is running and accessible
2. Firewall allows outbound connections
3. Authentication credentials are valid
4. URL/endpoint is correct
</Warning>

```bash
# Test connectivity
curl -v https://mcp-server.example.com/events

# Test with auth
curl -v -H "Authorization: Bearer token" \
  https://mcp-server.example.com/events
```

### Authentication Errors

```bash
# Verify token is valid
qai config test-transport audit --debug

# Check token expiration for OAuth2
curl https://auth.example.com/token/info \
  -H "Authorization: Bearer token"
```

### Timeout Issues

Increase timeouts for slow networks:

```bash
export QAI_AUDIT_MCP_TIMEOUT_MS=60000
qai audit ...
```
