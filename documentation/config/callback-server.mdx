# Callback Server Configuration

Configure the Interactive Payload Injection (IPI) callback server for receiving and processing payload responses from target applications.

## Overview

The callback server enables qai to receive real-time responses from injected payloads in target applications. This allows:

- **Live payload verification**: Confirm payload execution in real applications
- **Dynamic response handling**: Adapt future payloads based on responses
- **Evidence collection**: Capture proof of vulnerability exploitation
- **Two-way communication**: Establish bidirectional channels with targets

## Prerequisites

- qai IPI module enabled
- Network accessibility from target application to callback server
- Configured notification/bridge endpoint
- Firewall rules allowing inbound connections

## Callback Server Architecture

The callback server consists of:

1. **HTTP Listener**: Receives incoming callback requests
2. **Authentication Handler**: Validates bridge tokens
3. **Response Parser**: Processes payload responses
4. **Event Manager**: Routes responses to payload handlers

## Starting the Callback Server

### Default Configuration

Start with default settings:

```bash
# Start callback server (port 8888 by default)
qai ipi listener
```

Output:
```
Starting IPI Callback Server...
✓ Server listening on http://localhost:8888
✓ Webhook endpoint: /webhooks/payload
✓ Health check: /health
✓ Ready to receive callbacks
```

### Custom Port and Binding

Configure server address and port:

```bash
# Listen on custom port
qai ipi listener --port 9000

# Listen on specific interface
qai ipi listener --host 192.168.1.100 --port 8888

# Listen on all interfaces (production)
qai ipi listener --host 0.0.0.0 --port 8888
```

### Configuration File

Define settings in `qai.json`:

```json
{
  "modules": {
    "ipi": {
      "listener": {
        "host": "0.0.0.0",
        "port": 8888,
        "enable_ssl": true,
        "ssl_cert": "/etc/qai/callback.crt",
        "ssl_key": "/etc/qai/callback.key",
        "notification_timeout_ms": 30000,
        "max_payload_size": 10485760,
        "rate_limit": 100
      }
    }
  }
}
```

<Note>
For production deployments, always use HTTPS with proper certificates to protect payload data in transit.
</Note>

## Notification URLs

The callback server sends notifications to your application via webhook URLs.

### Configuring Notification URLs

Set the URL where qai sends callbacks:

```bash
# Via environment variable
export QAI_IPI_NOTIFY_URL=https://your-app.example.com/qai/callbacks

# Via CLI flag
qai audit --notify-url https://your-app.example.com/qai/callbacks

# Via configuration file
{
  "modules": {
    "ipi": {
      "notify_url": "https://your-app.example.com/qai/callbacks"
    }
  }
}
```

### Notification Payload Format

Callback notifications are sent as POST requests with JSON payloads:

```json
{
  "timestamp": "2024-03-22T10:30:15Z",
  "campaign_id": "campaign_abc123",
  "payload_id": "payload_def456",
  "status": "success",
  "response_data": {
    "method": "POST",
    "url": "/api/user/profile",
    "status_code": 200,
    "headers": {
      "content-type": "application/json"
    },
    "body": "{\"user_id\": 123, \"role\": \"admin\"}",
    "execution_time_ms": 45
  },
  "verification": {
    "payload_reflected": true,
    "command_executed": false,
    "access_gained": false
  }
}
```

## Bridge Token Authentication

The IPI callback listener runs as a separate process from the main qai web server. When a hit arrives, the listener notifies the web server via an internal HTTP POST so the web UI can update in real time via WebSocket. This internal communication is authenticated with a shared bridge token.

### How It Works

The bridge token is auto-generated the first time either process starts. It is stored at `~/.qai/bridge.token` (32 hex characters, file permissions `0600` on POSIX). Both processes read from the same file — no manual configuration is needed.

### Notify URL

The `--notify-url` flag on `qai ipi listen` controls where hit notifications are sent:

```bash
# Default: notifies the qai web server at http://127.0.0.1:8899
qai ipi listen --port 8080

# Custom notify URL (if web server is on a different port)
qai ipi listen --port 8080 --notify-url http://127.0.0.1:9000
```

The listener includes the bridge token in the `Authorization` header of its internal POST to the web server. If the tokens don't match (e.g., stale token file), the notification is rejected and the hit is still recorded in the IPI database but won't appear in the web UI until the page is refreshed.

<Tip>
If WebSocket hit notifications stop working, delete `~/.qai/bridge.token` and restart both the web server and the listener. A fresh token will be generated automatically.
</Tip>

## Callback Payload Endpoints

### Webhook Endpoints

The callback server exposes these endpoints:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/webhooks/payload` | POST | Primary callback endpoint |
| `/webhooks/verify` | POST | Verify payload execution |
| `/webhooks/evidence` | POST | Submit proof of exploitation |
| `/health` | GET | Health check |

### Example Payload Callback

```bash
curl -X POST https://your-server.example.com/webhooks/payload \
  -H "Content-Type: application/json" \
  -H "X-Bridge-Token: sk-ipi-..." \
  -d '{
    "campaign_id": "campaign_abc123",
    "payload_id": "payload_def456",
    "timestamp": "2024-03-22T10:30:15Z",
    "response": {
      "status": "executed",
      "output": "injection_point_confirmed",
      "evidence": "error message revealed database version: MySQL 5.7.25"
    }
  }'
```

## Setting Up for External Access

### Network Configuration

Configure callback server for external accessibility:

```bash
# Allow all interfaces (for internal networks only)
qai ipi listener --host 0.0.0.0 --port 8888

# Or bind to specific external IP
qai ipi listener --host 203.0.113.42 --port 8888
```

### HTTPS Setup

Enable TLS for secure callback transmission:

```json
{
  "modules": {
    "ipi": {
      "listener": {
        "host": "0.0.0.0",
        "port": 8888,
        "enable_ssl": true,
        "ssl_cert": "/etc/qai/certs/callback.crt",
        "ssl_key": "/etc/qai/certs/callback.key",
        "ssl_protocols": ["TLSv1.2", "TLSv1.3"]
      }
    }
  }
}
```

Generate self-signed certificate for testing:

```bash
openssl req -x509 -newkey rsa:4096 -keyout callback.key -out callback.crt -days 365 -nodes
```

### DNS and Reverse Proxy

Set up public DNS and reverse proxy:

```nginx
# Nginx reverse proxy configuration
upstream qai_callback {
    server localhost:8888;
}

server {
    listen 443 ssl http2;
    server_name callback.your-domain.com;

    ssl_certificate /etc/letsencrypt/live/callback.your-domain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/callback.your-domain.com/privkey.pem;

    location / {
        proxy_pass http://qai_callback;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## Response Handling

### Automatic Response Processing

The callback server automatically processes responses:

```json
{
  "response_handlers": [
    {
      "pattern": "error_message",
      "action": "log_evidence",
      "next_payload": "extract_version"
    },
    {
      "pattern": "command_executed",
      "action": "escalate_severity",
      "next_payload": "read_sensitive_file"
    }
  ]
}
```

### Custom Response Processors

Write custom handlers for specialized processing:

```python
from qai.ipi.server import ResponseHandler

class CustomResponseHandler(ResponseHandler):
    def handle(self, payload_response):
        # Custom processing logic
        if payload_response.get('status') == 'executed':
            # Log evidence
            self.log_evidence(payload_response)

            # Trigger next payload
            self.queue_payload('next_payload_id')

            # Update campaign status
            self.update_campaign_status('escalated')
```

## Monitoring and Logging

### View Callback History

```bash
# Show recent callbacks
qai ipi callbacks --limit 50

# Filter by campaign
qai ipi callbacks --campaign campaign_abc123

# Export callbacks for analysis
qai ipi callbacks --export csv > callbacks.csv
```

### Real-time Callback Monitoring

Monitor callbacks as they arrive:

```bash
# Watch live callbacks
qai ipi callbacks --follow

# Output:
# [10:30:15] campaign_abc123 | payload_def456 | status:success | 200ms
# [10:30:16] campaign_abc123 | payload_ghi789 | status:failed | 150ms
```

Configure logging for callback events:

```json
{
  "logging": {
    "ipi": {
      "level": "debug",
      "file": "/var/log/qai/ipi-callbacks.log",
      "rotation": "daily",
      "retention_days": 30
    }
  }
}
```

## Security Considerations

### Authentication

Always use bridge tokens for authentication:

<Warning>
Never expose callback endpoints without authentication. Always use:
- Bridge token validation
- HTTPS/TLS encryption
- IP whitelisting if possible
</Warning>

### Rate Limiting

Configure rate limits to prevent abuse:

```json
{
  "modules": {
    "ipi": {
      "listener": {
        "rate_limit": 100,
        "burst_limit": 200,
        "rate_window_seconds": 60
      }
    }
  }
}
```

### Firewall Rules

Restrict callback server access:

```bash
# Only allow from specific networks
sudo ufw allow from 10.0.0.0/8 to any port 8888
sudo ufw allow from 203.0.113.0/24 to any port 8888
```

## Troubleshooting

### Server Won't Start

```bash
# Check port is not in use
lsof -i :8888

# Try different port
qai ipi listener --port 9000
```

### Callbacks Not Received

```bash
# Verify notification URL is correct
echo $QAI_IPI_NOTIFY_URL

# Test connectivity to notification URL
curl https://your-app.example.com/qai/callbacks

# Check firewall rules
sudo firewall-cmd --list-all
```

### Authentication Failures

```bash
# Verify bridge token is set
qai config list-providers | grep ipi

# Generate new token
qai ipi generate-token --campaign campaign_abc123

# Test authentication
curl -H "X-Bridge-Token: sk-ipi-..." \
  https://callback.example.com/health
```
