---
title: Inference Target Configuration
description: Configure the demonstrated OpenAI-compatible target and OS-keyring credential
---

The proxy and manual operator workflows do not require a model credential. The driven-inference
workflow uses a persisted target with an OpenAI-compatible endpoint, exact model pin, fixed
generation parameters, and the name of an OS-keyring credential.

## Store the credential

```bash
ctpf config set-credential lab-inference
```

The command prompts with masked input and stores the value in Windows Credential Manager,
macOS Keychain, or Linux Secret Service. API keys are never read from environment variables,
configuration files, target metadata, or source code.

<Warning>
If no secure OS-backed keyring is available, configure one before running driven inference.
The harness fails closed rather than falling back to plaintext or environment variables.
</Warning>

Legacy plaintext credentials can be migrated once with:

```bash
ctpf config import-legacy-credentials
```

## Register an OpenAI-compatible target

```bash
ctpf targets add "Lab Inference" http://inference-server:11434/v1 \
  --type inference \
  --meta driver=openai-compatible \
  --meta model=qwen3.5:4b \
  --meta credential=lab-inference \
  --meta max_tokens=512 \
  --meta temperature=0 \
  --meta seed=0 \
  --meta reasoning_effort=none
```

The endpoint and model are exact experiment pins. The `credential` value is only a keyring entry
name; secret material is not stored in SQLite or evidence bundles.

## Claude Code runtime targets

The demonstrated external-runtime adapter does not use a provider SDK or API-key setting. It uses
Claude Code's runtime-managed secure login, an exact model ID, a minimum supported runtime version,
and a minimal non-secret process environment.
