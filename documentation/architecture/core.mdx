The `ctpf.core` package provides the operational backbone used by the CLI and demonstrated target
adapters. It does not own scenario semantics or the experiment evidence directory.

## Operational SQLite state

CTPF uses SQLite with WAL mode and foreign keys at `~/.ctpf/ctpf.db`. Current tables support targets,
settings, runs, findings, evidence references, and proxy/audit records. Upgraded databases may also
contain historical tables for removed modules.

| Model | Current role |
| --- | --- |
| Target | Persist a demonstrated endpoint/runtime identity and non-secret metadata |
| Run | Represent retained operational activity; not the condition manifest for `ctpf experiment` |
| Finding | Store secondary-library or other operational findings |
| Evidence | Store operational evidence references; not scenario bundle content |
| Setting | Store non-secret configuration values |

Experiment traces, transcripts, mutation records, effects, manifests, and bundles are written to
the external `--output-root`, not SQLite.

## Target profiles

Driven experiments load a persisted target and validate it into one of the demonstrated profiles:

- an OpenAI-compatible inference endpoint with exact model/generation pins and a credential name;
- a Claude Code agent-runtime target with an exact model, runtime constraints, and runtime-managed
  authentication; or
- an MCP server target used by the supporting proxy/MCP layer.

Target ID prefixes must contain at least eight characters and be unambiguous.

## Configuration precedence

Non-secret configuration resolves in this order:

1. CLI flags
2. Explicitly mapped environment variables
3. SQLite settings managed through `ctpf config`
4. `~/.ctpf/config.yaml`
5. Built-in defaults

Credential resolution is a separate boundary: OS keyring → error. API keys are never read from
environment variables, configuration files, target metadata, or source code.

## Credential management

The keyring service is `CTPF`.

- `ctpf config set-credential NAME` prompts with masked input.
- `ctpf config delete-credential NAME` removes a keyring entry.
- `ctpf config list-providers` reports configured credential names.
- `ctpf config import-legacy-credentials` performs the one-time plaintext migration path.

## LLM boundary

`ctpf.core.llm` defines the provider-neutral client protocol used by driven inference.
`ctpf.core.llm_litellm` supplies the demonstrated OpenAI-compatible implementation and returns
normalized response objects. The Claude Code adapter is separate and relies on the runtime's secure
login rather than an embedded provider credential.

## Frozen audit support

Framework and mitigation resolvers remain available to the frozen audit library. They do not score
controlled CTPF experiments and are not part of the method contribution.

## Historical compatibility

Schema columns, tables, migrations, bundle helpers, and reset/deletion logic for removed modules
remain so existing databases can be opened, backed up, reset, and cleaned safely. Retaining those
data contracts does not restore their product writers.

The public `export_run_bundle()` compatibility helper is documented at
[Operational Run Bundle](/exports/json-schema). It is distinct from controlled experiment evidence
bundles.
