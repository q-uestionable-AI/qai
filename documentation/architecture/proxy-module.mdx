---
title: Proxy Architecture
description: MCP capture, controlled intervention, original preservation, and replay
---

`ctpf.proxy` is supporting observation and intervention infrastructure for controlled MCP traffic.
It preserves protocol evidence needed by CTPF experiments and also exposes a manual TUI, replay,
export, and inspection surface.

## Module structure

```text
proxy/
├── intercept.py          # Hold, forward, modify, or drop messages
├── pipeline.py           # Bidirectional processing and capture
├── runtime.py            # Loopback proxy runtime used by experiments
├── session_store.py      # JSON session persistence
├── correlation.py        # Request/response correlation
├── replay.py             # Replay captured client messages
├── models.py             # ProxyMessage, ProxySession, and intercept state
├── adapters/             # stdio, SSE, and Streamable HTTP transports
├── tui/                  # Textual manual operator interface
├── exporting/            # Session JSON export
├── cli.py                # start, replay, export, and inspect commands
└── mapper.py             # Optional operational database mapping helper
```

There is no `ProxyAdapter` or orchestrator integration in the current package.

## Runtime flow

1. A controlled client connects to the loopback-facing proxy endpoint or TUI path.
2. The runtime connects upstream to the declared MCP server or packaged fixture.
3. The pipeline wraps every JSON-RPC message with sequence, direction, transport, and correlation
   metadata.
4. Passthrough forwards messages immediately; intercept mode holds them for forward, modify, or
   drop decisions.
5. A modified message stores the pre-change JSON-RPC body in `original_raw` before forwarding.
6. The completed session is serialized for experiment parsing, manual inspection, or replay.

## Experiment boundary

Controlled experiments configure the loopback runtime directly and apply narrow, scenario-owned
mutation rules. The proxy supplies protocol capture and original preservation; it does not decide
the scenario's authority semantics, score the trust transition, or interpret the result.

## Session model

`ProxyMessage` records direction, transport, JSON-RPC identity/method, sequence, correlation,
modification state, the forwarded message, and the original when changed. `ProxySession` stores the
ordered messages plus connection metadata and start/end timestamps.

## Replay boundary

`ctpf proxy replay` sends captured client-to-server messages to a live authorized server and records
new responses. This is server interaction replay, not full agent counterfactual replay and not a
CTPF experiment classification by itself.

## Local binding invariant

Every CTPF-owned HTTP listener binds to `127.0.0.1`. Remote MCP targets may be contacted as declared
upstream endpoints, but the local proxy never listens on an external interface.
