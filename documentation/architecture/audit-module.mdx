## Audit Module

The Audit module discovers security scanners and orchestrates assessment runs. It normalizes findings across diverse scanner types and generates reports in multiple formats.

### Module Structure

```
audit/
├── orchestrator.py      # Run coordination and scanner orchestration
├── scanner/             # Scanner discovery and execution
├── reporting/           # Report generation (JSON, SARIF, HTML, etc.)
├── cli.py               # Typer command interface
├── adapter.py           # Module adapter for platform integration
└── mapper.py            # Framework mapping (ATT&CK, CIS, etc.)
```

### Scanner System

The Audit module discovers and manages scanners dynamically.

**Scanner Discovery**

On startup, the module scans configured directories for scanner plugins:

```bash
qai audit discover --list-all
```

Scanner directories:
- Built-in: `src/q_ai/audit/scanner/`
- User: `~/.qai/scanners/`
- Custom: `QAI_SCANNER_PATH`

**Scanner Interface**

Each scanner implements:

```python
class Scanner:
    id: str                              # Unique identifier
    name: str                            # Display name
    description: str
    version: str
    supported_targets: [str]             # url, host, port, etc.
    frameworks: [str]                    # MITRE ATT&CK, CIS, etc.

    async def validate_target(target: Target) -> bool
    async def execute(target: Target) -> [Finding]
    async def health_check() -> bool
```

<Note>
Scanners run in isolated processes. If a scanner crashes, it doesn't affect other scanners or the audit run.
</Note>

### Audit Execution Flow

```
User initiates audit
    ↓
Run created (status: PENDING)
    ↓
FrameworkResolver loads configured frameworks
    ↓
For each target:
    ├─ Scanner validation
    ├─ Parallel scanner execution
    ├─ Result normalization
    └─ Finding storage
    ↓
Run status: COMPLETED or PARTIAL
    ↓
User triggers report generation
```

**Parallel Execution**

Multiple scanners execute concurrently:

```python
# Orchestrator spawns parallel tasks
results = await asyncio.gather(
    scanner_1.execute(target),
    scanner_2.execute(target),
    scanner_3.execute(target),
    return_exceptions=True
)
```

### Finding Normalization

Scanners produce diverse output formats. The audit module normalizes findings into the core Finding model.

**Scanner Output Example**

```json
{
  "scan_id": "scan-12345",
  "scan_date": "2024-01-15T10:30:00Z",
  "results": [
    {
      "severity": "high",
      "message": "Unencrypted HTTP endpoint detected",
      "affected_resource": "example.com",
      "cve": "CVE-2024-1234",
      "remediation": "Enable HTTPS"
    }
  ]
}
```

**Normalization Steps**

1. **Severity Mapping**: Normalize scanner severity (critical, high, medium, low, info, unknown) to core enum
2. **Framework Mapping**: Map scanner findings to framework technique IDs using FrameworkResolver
3. **Evidence Extraction**: Store raw scanner output and structured fields
4. **Deduplication**: Merge duplicate findings within same run

**Mapping Configuration**

Define scanner→framework mappings in `scanner-mappings.yaml`:

```yaml
scanners:
  nmap:
    frameworks:
      - name: mitre-attack
        mappings:
          - finding: "Open port"
            technique_id: T1046
          - finding: "Service enumeration"
            technique_id: T1526
  trivy:
    frameworks:
      - name: cis-benchmarks
        mappings:
          - pattern: "^CVE-"
            technique_id: CIS-x.y.z
```

### Reporting Pipeline

You can generate reports in multiple formats from audit results.

**Available Formats**

| Format | Extension | Use Case |
|--------|-----------|----------|
| JSON | .json | Machine parsing, data exchange |
| SARIF | .sarif | IDE/SIEM integration, vulnerability tools |
| HTML | .html | Executive dashboard, visual review |
| NDJSON | .ndjson | Streaming logs, real-time analysis |
| CSV | .csv | Spreadsheet analysis, pivot tables |

**Report Generation**

```bash
# Generate all reports
qai audit report --run-id <id> --format all --output-dir ./reports

# Generate specific format
qai audit report --run-id <id> --format sarif --output ./findings.sarif

# Stream NDJSON
qai audit report --run-id <id> --format ndjson --follow
```

### Framework Integration

<Tip>
The audit module references the core FrameworkResolver to ensure consistent framework mapping across all modules.
</Tip>

**Framework Resolution Process**

1. Load configured framework definitions
2. For each finding, determine potential technique matches
3. Apply scanner-specific mapping rules
4. Store framework_id in Finding model
5. Include technique metadata in reports

**Multi-Framework Support**

Run audit against multiple frameworks:

```bash
qai audit run \
  --scanner nmap \
  --frameworks mitre-attack,cis-benchmarks \
  --target example.com
```

Results include mappings to all configured frameworks:

```json
{
  "finding": "Open port 22",
  "mappings": [
    {
      "framework": "mitre-attack",
      "technique_id": "T1046",
      "technique_name": "Network Service Discovery"
    },
    {
      "framework": "cis-benchmarks",
      "benchmark_id": "CIS-x.y.z",
      "benchmark_name": "Ensure SSH protocol version 2"
    }
  ]
}
```

### Run Lifecycle

**Creating a Run**

```bash
qai audit run create \
  --name "Q1 Assessment" \
  --target example.com \
  --scanners nmap,trivy \
  --framework mitre-attack
```

**Monitoring Progress**

```bash
# Poll status
qai audit run status --run-id <id>

# Watch live (WebSocket)
qai audit run watch --run-id <id>
```

**Run Status Transitions**

```
PENDING → RUNNING → COMPLETED
              ↓
            PARTIAL (if some scanners fail)
              ↓
            FAILED (if critical failure)

User can cancel at any time → CANCELLED
```

**Accessing Results**

```bash
# List findings
qai audit findings list --run-id <id>

# Get specific finding with evidence
qai audit findings get --finding-id <id> --include-evidence

# Export to file
qai audit report --run-id <id> --format json --output results.json
```

### Performance Considerations

**Scan Timeout**

Each scanner has a configurable timeout (default: 5 minutes):

```bash
qai audit run --scanner nmap --timeout 600s
```

**Parallel Limits**

The orchestrator limits parallel scanners to avoid resource exhaustion:

```yaml
# audit-config.yaml
parallel-scanners: 4
scanner-timeout-seconds: 300
```

**Result Caching**

Audit caches scanner discovery results (5 minute TTL):

```bash
qai audit discover --no-cache  # Bypass cache
```

### Custom Scanners

Extend audit with custom scanners:

```python
# ~/.qai/scanners/my_scanner.py
from q_ai.audit.scanner import Scanner, Finding

class MyScanner(Scanner):
    id = "my-scanner"
    name = "Custom Scanner"

    async def execute(self, target):
        findings = []
        # Your scanning logic
        for item in scan_results:
            findings.append(Finding(
                title=item.title,
                description=item.desc,
                severity="high",
                framework_id="T1234"
            ))
        return findings
```

Register with:

```bash
qai audit discover --register-path ~/.qai/scanners
```

### Integration with Other Modules

The Audit module outputs are consumed by:

- **Chain Module**: References audit findings in attack chain templates
- **IPI Module**: Audit results inform adversarial document generation
- **Web UI**: Displays findings in unified dashboard

<Note>
All findings include `run_id` linking to the core Run model, enabling cross-module correlation.
</Note>
