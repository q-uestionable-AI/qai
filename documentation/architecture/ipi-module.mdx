## IPI Module

The IPI (Instant Payload Injection) module generates adversarial documents in 7 formats, deploys them via multiple channels, and tracks user interactions in real-time via callback notifications.

### Module Structure

```
ipi/
├── generators/          # Format-specific document generators
│   ├── pdf.py
│   ├── docx.py
│   ├── image.py
│   ├── markdown.py
│   ├── html.py
│   ├── ics.py
│   └── eml.py
├── models.py            # Document, campaign, and hit models
├── listener.py          # Callback listener for hit notifications
├── server.py            # HTTP API for document serving/deployment
├── generate_service.py   # Generator orchestration
├── guidance_builder.py   # Deployment playbook generation
├── db.py                # IPI-specific database tables
├── cli.py               # Typer command interface
├── adapter.py           # Module adapter for platform integration
├── api.py               # HTTP endpoints
├── ui.py                # Web UI routes
└── templates/           # Document templates
```

### Supported Document Formats

| Format | Extension | Use Cases | Techniques |
|--------|-----------|-----------|---|
| PDF | .pdf | Reports, forms, resumes | Embedded scripts, metadata exploits |
| DOCX | .docx | Proposals, contracts | Macros, external objects |
| Image | .png, .jpg | Screenshots, diagrams | Metadata, EXIF data, steganography |
| Markdown | .md | Documentation, readmes | Embedded links, script tags |
| HTML | .html | Web pages, emails | JavaScript, form hijacking |
| ICS | .ics | Calendar invites | Malicious event data |
| EML | .eml | Email messages | Phishing content, malicious links |

### Document Model

```python
Document {
    id: str (uuid)
    campaign_id: str (foreign key → Campaign)
    format: str (pdf, docx, image, markdown, html, ics, eml)
    filename: str
    title: str
    description: str
    content: bytes (binary document data)
    metadata: dict
    hiding_techniques: [str]
    deployment_urls: [str]
    created_at: datetime
}

Campaign {
    id: str (uuid)
    run_id: int (foreign key → Run)
    name: str
    description: str
    objective: str (social engineering, payload delivery, etc.)
    documents: [Document]
    deployments: [Deployment]
    hits: [Hit]
    started_at: datetime
    completed_at: datetime
    status: CampaignStatus (DRAFT, ACTIVE, COMPLETED, ARCHIVED)
}
```

### Generator Architecture

The generator system provides a base interface with format-specific implementations.

**Base Generator Protocol**

```python
class Generator:
    """Abstract interface for document generators."""

    format_id: str
    supported_techniques: [str]

    async def generate(
        self,
        title: str,
        content: str,
        hiding_techniques: [str] = []
    ) -> Document:
        """Generate adversarial document."""

    async def validate(self, document: Document) -> bool:
        """Verify document viability."""

    def get_technique_info(self, technique_id: str) -> dict:
        """Get details about hiding technique capabilities."""
```

```yaml

  - id: "steganography"
    name: "Steganographic Hiding"
    formats: ["image", "pdf"]

    name: "Exfiltration Callback"
```

**Using Techniques**


### Document Generation Pipeline

**Generation Flow**

```
User requests document generation
    ↓
Campaign created (status: DRAFT)
    ↓
For each requested format:
    ├─ Load template
    ├─ Apply hiding techniques
    ├─ Embed callback URL
    ├─ Generate document
    └─ Validate document integrity
    ↓
Campaign status: READY
    ↓
User reviews and deploys
```

**Generation API**

```bash
# Create campaign
qai ipi campaign create \
  --name "Social Engineering Test" \
  --objective "credential-harvesting"

# Generate documents
qai ipi generate \
  --campaign-id <id> \
  --formats docx,pdf,html \
  --techniques macro-injection,external-objects \
  --callback-url https://callback.example.com

# List generated documents
qai ipi documents list --campaign-id <id>

# Download specific document
qai ipi documents get \
  --document-id <id> \
  --output ~/Downloads/report.docx
```

### Callback Listener Architecture

The listener receives hit notifications when documents are accessed.

**Hit Model**

```python
Hit {
    id: str (uuid)
    document_id: str (foreign key → Document)
    campaign_id: str (foreign key → Campaign)
    timestamp: datetime
    client_ip: str
    user_agent: str
    metadata: dict
    confidence: float
    evidence: dict
}
```

**Callback Server**

```python
# HTTP endpoint that documents contact
@app.post("/callback/{document_id}")
async def handle_hit(document_id: str, request: Request):
    """Process document hit notification."""

    # Record hit
    hit = Hit.create(
        document_id=document_id,
        client_ip=request.client.host,
        user_agent=request.headers.get("user-agent"),
        timestamp=datetime.now(),
        metadata={
            "referrer": request.headers.get("referer"),
            "headers": dict(request.headers)
        }
    )

    # Update campaign status
    campaign = Campaign.from_document(document_id)
    campaign.status = "HIT_DETECTED"
    campaign.save()

    # Notify Web UI via WebSocket
    await broadcast_hit_notification(hit)

    return {"status": "recorded"}
```

**WebSocket Hit Notifications**

Real-time hit notifications via WebSocket:

```javascript
// Web UI client
const ws = new WebSocket("wss://qai.example.com/ws/campaign/campaign-uuid");

ws.onmessage = (event) => {
    const hit = JSON.parse(event.data);
    console.log(`Document hit: ${hit.document_id} from ${hit.client_ip}`);
    updateCampaignUI(hit);
};
```

### Deployment Methods

Documents are deployed through multiple channels.

**Deployment Modes**

| Mode | Channel | Setup |
|------|---------|-------|
| HTTP Link | Email/Messaging | Host documents on callback server |
| Direct Email | SMTP | Send via SMTP server |
| File Share | Network share | Deploy to accessible share |
| Physical Media | USB/CD | Manual deployment |

**HTTP Deployment**

```bash
qai ipi deploy \
  --campaign-id <id> \
  --method http \
  --callback-url https://callback.example.com
```

Documents hosted at:
- `https://callback.example.com/docs/{document_id}`

**Email Deployment**

```bash
qai ipi deploy \
  --campaign-id <id> \
  --method email \
  --recipient-file targets.csv \
  --smtp-server smtp.example.com \
  --from-address phishing@example.com
```

Recipients CSV format:
```
email,name
john.smith@example.com,John Smith
jane.doe@example.com,Jane Doe
```

### Campaign Lifecycle

**Campaign Status Flow**

```
DRAFT (documents being created)
    ↓
READY (all documents generated, awaiting deployment)
    ↓
ACTIVE (documents deployed, hits being tracked)
    ↓
COMPLETED (campaign ended by user)
    ↓
ARCHIVED (historical data retained)
```

**Campaign Monitoring**

```bash
# Get campaign status
```

### IPI Database Schema

```sql
-- Campaigns
CREATE TABLE ipi_campaigns (
  id TEXT PRIMARY KEY,
  run_id INTEGER REFERENCES runs(id),
  name TEXT,
  objective TEXT,
  status TEXT,
  created_at TIMESTAMP
);

-- Generated documents
CREATE TABLE ipi_documents (
  id TEXT PRIMARY KEY,
  campaign_id TEXT REFERENCES ipi_campaigns(id),
  format TEXT,
  filename TEXT,
  title TEXT,
  content BLOB,
  created_at TIMESTAMP
);

-- Document techniques
CREATE TABLE ipi_document_techniques (
  id INTEGER PRIMARY KEY,
  document_id TEXT REFERENCES ipi_documents(id),
  technique_id TEXT
);

-- Hit tracking
CREATE TABLE ipi_hits (
  id TEXT PRIMARY KEY,
  document_id TEXT REFERENCES ipi_documents(id),
  campaign_id TEXT REFERENCES ipi_campaigns(id),
  timestamp TIMESTAMP,
  client_ip TEXT,
  user_agent TEXT,
  metadata BLOB
);
```

### Integration with Other Modules

<Note>
IPI campaigns reference Audit findings to determine appropriate document content, and correlate hits with Inject campaign results for comprehensive threat assessment.

**Cross-Module Correlation**

- **Audit Findings**: Use vulnerability information to inform document content
- **Inject Campaigns**: Compare success rates across injection and document delivery
- **Chain Execution**: Include document hits as evidence in attack chain analysis

### Performance and Scalability

**Document Generation**

Generation is CPU-bound. Configure:

```yaml
ipi:
  max-concurrent-generations: 4
  timeout-seconds: 60
```

**Hit Tracking**

Hits are logged and broadcast via WebSocket:

```yaml
ipi:
  hit-buffer-size: 100
  hit-flush-interval-ms: 1000
  websocket-broadcast-enabled: true
```

**Database Performance**

The IPI database uses indices for common queries:

```sql
CREATE INDEX idx_hits_campaign ON ipi_hits(campaign_id, timestamp);
CREATE INDEX idx_hits_document ON ipi_hits(document_id);
CREATE INDEX idx_documents_campaign ON ipi_documents(campaign_id);
```
