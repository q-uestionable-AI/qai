## Platform Architecture

The qai platform is built as a modular, layered system with seven specialized security assessment modules unified through a shared core infrastructure. Each module operates independently while leveraging common data storage, configuration, and runtime services.

### Core Architecture Layers

<div style={{display: 'grid', gridTemplateColumns: '1fr', gap: '1rem', marginTop: '1rem'}}>

**Web UI Layer**
- FastAPI application server with HTMX for dynamic interactions
- Jinja2 templating for server-side rendering
- WebSocket channels for real-time campaign notifications
- Browser-based access to audit results, findings, and campaign management

**Module Layer**
- Seven security modules with specialized functions (Audit, Proxy, Inject, Chain, IPI, CXP, RXP)
- Consistent adapter pattern for module integration
- Shared CLI infrastructure via Typer subcommands
- Independent orchestration and reporting pipelines

**Shared Core Layer**
- SQLite database with WAL mode and foreign key enforcement
- Centralized configuration management (YAML, environment, keyring, database)
- Framework resolver for mapping assessment categories
- Provider registry for cloud and custom services
- LLM interface for guidance generation

**Data Layer**
- SQLite persistent storage with schema versioning
- Run lifecycle management and audit trails
- Finding storage with structured severity and status tracking
- Evidence correlation and field-level metadata

</div>

### Module System

You interact with each module through dedicated subsystems:

**Audit Module** (`audit`)
- Discovers and executes security scanners
- Maps findings to frameworks (MITRE ATT&CK, CIS, etc.)
- Exports results in multiple formats (JSON, SARIF, HTML, NDJSON, CSV)

**Proxy Module** (`proxy`)
- Intercepts and records MCP message traffic
- Provides terminal UI for session review
- Captures and exports interaction evidence

**Inject Module** (`inject`)
- Executes multi-provider LLM campaigns
- Normalizes responses across providers
- Scores injection attempts with evidence preservation

**Chain Module** (`chain`)
- Executes YAML-defined attack chains
- Analyzes blast radius of sequential steps
- Generates detection rules (Sigma, Wazuh format)

**IPI Module** (`ipi`)
- Generates adversarial documents in 7 formats
- Deploys documents via HTTP/email
- Captures and correlates document hits
- Builds deployment playbooks with remediation guidance

**CXP Module** (`cxp`)
- Generates assistant instruction files
- Validates prompt injection rules
- Organizes techniques by framework objectives

**RXP Module** (`rxp`)
- Validates responses with embedding models
- Builds domain-specific detection profiles
- Generates interpretive bands for anomaly detection

### Unified Command Interface

All modules share a consistent CLI structure:

```
qai <module> <subcommand> [options]
```

Example operations:

```bash
qai audit discover                    # Find available scanners
qai audit run --scanner <name>        # Execute audit
qai inject campaign create --name <n> # Start new campaign
qai ipi generate --format pdf         # Generate adversarial doc
qai chain execute --template <t>      # Run attack chain
```

### Data Flow Through Modules

<Note>
Each module maintains its own database tables while sharing the core Run model. This allows independent scaling while preserving audit relationships.
</Note>

1. **Run Creation**: User initiates action in Web UI or CLI
2. **Module Processing**: Dedicated module processes the request using domain-specific logic
3. **Data Persistence**: Results stored in SQLite with Run reference
4. **Cross-Module Reference**: Chain/IPI can reference Audit findings; Inject correlates with IPI hits
5. **Reporting**: Web UI aggregates across modules for unified dashboard view

### Configuration Hierarchy

Configuration flows through a precedence chain:

```
CLI arguments
  ↓ (override)
Environment variables
  ↓ (override)
Database settings
  ↓ (override)
YAML configuration
  ↓ (override)
Defaults
```

<Tip>
Use `qai config show` to view the merged configuration at any precedence level.
</Tip>

### Shared Infrastructure Components

**FrameworkResolver**
- Loads framework definitions from `frameworks.yaml`
- Maps assessment categories to framework technique IDs
- Enables cross-module framework consistency

**RunGuidance System**
- Generates structured deployment playbooks
- Organizes mitigation guidance by framework
- Used by IPI and CXP modules for output generation

**Provider Registry**
- Manages cloud providers (AWS, Azure, GCP, Kubernetes)
- Supports custom provider plugins
- Enables module-agnostic provider access

**Bridge Token**
- Shared secret for IPI↔WebUI authentication
- Enables secure callback notification routing
- Generated and managed by core infrastructure

### WebSocket Integration

<Warning>
WebSocket connections require valid bridge token authentication. The platform automatically manages token generation and refresh.
</Warning>

Real-time module interactions use WebSocket for:
- Campaign progress notifications from IPI module
- Finding updates from Audit module
- Chain execution events

The Web UI automatically subscribes to relevant channels based on active campaigns. Each WebSocket message includes the Run ID for correlation.

### Background Task Execution

The orchestrator manages long-running operations:

```
CLI Command / Web Request
  ↓
WorkflowRunner.execute()
  ↓
Background task spawned (async)
  ↓
Run status updated → RUNNING
  ↓
Module processes asynchronously
  ↓
Completion callback → Run status updated
```

The Web UI polls Run status or receives WebSocket notifications to display live progress.
