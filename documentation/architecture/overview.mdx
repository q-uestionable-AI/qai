CTPF is a research project for testing Capability Trust Propagation Failure. The Python package is
its evidence-first reference harness: it coordinates controlled conditions and preserves the
experimental record. It is not the thesis itself or a generic scanner, proxy product, red-team
platform, or workflow orchestrator.

## Responsibility layers

| Layer | Responsibility | Location |
| --- | --- | --- |
| Research thesis | Define the capability-trust question and publication claims | Research program outside the repository |
| Human governance | Sign policy/approvals; authorize targets, spend, science, and publication | `ctpf experiment govern` (TTY) |
| Agent-operable control | Untrusted machine lifecycle over exact contracts | `ctpf experiment control` (source `main`) |
| Stable harness | Condition isolation, protocol capture, trust-transition records, comparison, effect verification, and evidence integrity | Reusable `ctpf` package |
| Study instrument | Scenario task, prompt, fixture, intervention, authority semantics, and oracle expectations | Narrow scenario modules and tests |
| External system | Supply a model or agent runtime required by a named study | Registered target |
| External sandbox | Contain a full-shell caller | Operator deployment |
| Interpretation | Decide what the observation establishes | Human researcher |

## Package shape

| Area | Responsibility |
| --- | --- |
| `experiment.py` | Coordinate isolated conditions, targets, manifests, comparisons, and bundles; mount `control`/`govern` |
| `automation/` | Contracts, policy, approval, lifecycle service, and execution control (source `main`) |
| `kernel` | Trust-transition schemas, observations, trace parsing, scenario comparison, effect oracles, evidence bundles, and verifier |
| `proxy` | MCP capture, controlled intervention, original preservation, persistence, replay, and export |
| `mcp` | Async MCP connections, transports, discovery, and protocol models |
| `driven_inference.py` | Demonstrated OpenAI-compatible model/tool loop and transcript capture |
| `external_runtime.py` | Demonstrated external-runtime seam, currently Claude Code for cascade |
| `core` | SQLite state, targets, settings, OS-keyring credentials, models, and provider-neutral LLM contracts |
| `services` | Small database maintenance helpers |
| `audit` | Frozen secondary enumeration/scanner library; not a root CLI or method contribution |

The public operator entry point is `ctpf`. Its research command is `ctpf experiment`. On source
`main`, `control` is the autonomous surface and `govern` is the human authority surface. Proxy,
targets, stored operational records, configuration, and database commands support that work and are
not the default autonomous entry. See [Agent-Operable Lifecycle](/experiments/agent-operable).

## Experimental flow

1. The scenario declares its prompt, tools, mutation, trust transition, and expected effect.
2. The coordinator allocates series and condition-scoped run identities.
3. Baseline, manipulated, and hardened conditions run sequentially with isolated state.
4. The proxy preserves protocol messages and the original for every modified response.
5. Manual, driven-inference, or demonstrated external-runtime operation drives the controlled client.
6. Scenario parsing records invocation, response, persistence, later consumption, and effect
   separately.
7. Comparison fails closed when causal or effect evidence is missing.
8. Progress manifests preserve failures; bundles declare and hash copied artifacts.

## Packaged scenarios

### Cascade memo

The released `ctpf==0.14.0` workflow uses two sessions per condition and can require exact
write → artifact → later read → action → sink continuity. Demonstrated seams include manual Cursor,
OpenAI-compatible driven inference, a sequential two-model matrix, and Claude Code.

### Pattern 2

Unreleased source `main` contains a one-session direct response-to-effect scenario. It records the
status response, matching `apply_change` invocation, and independently verified sink effect. One
OpenAI-compatible acceptance series has been demonstrated; no Pattern 2 Claude Code result exists.

Shared engineering machinery across these scenarios is not scientific generality. Their prompts,
tools, trust semantics, parsers, and oracle rules remain scenario-specific.

## Target boundaries

Persisted targets identify one demonstrated execution seam:

- `inference` with `driver=openai-compatible`, an exact model and generation pins, and an OS-keyring
  credential name;
- `agent-runtime` with `driver=claude-code-cli`, an exact model/runtime pin, and runtime-managed
  authentication; or
- an MCP server target used by observation/proxy workflows.

There is no adapter registry, plugin system, workflow DAG, recipe language, or general
external-artifact ingestion contract.

## State and evidence

SQLite at `~/.ctpf/ctpf.db` stores targets, settings, and operational records. Experiment traces,
transcripts, mutations, effects, manifests, and bundles go to the operator-selected external
research directory. The two stores are intentionally distinct.

## Security and evidence invariants

- CTPF-owned local HTTP listeners bind to `127.0.0.1` only.
- API keys are stored and read only through the OS keyring.
- Modified responses preserve their originals.
- Condition state and external effects are run-scoped.
- Missing or contaminated evidence is not reconstructed.
- Experiment output directories must be outside Git checkouts.

## Removed and frozen surfaces

Web UI, Assist, RXP, Chain, Orchestrator, Imports, IPI, CXP, and Inject packages were removed.
Historical database tables may remain for migration and deletion compatibility. Audit remains
in-tree but frozen and secondary.
