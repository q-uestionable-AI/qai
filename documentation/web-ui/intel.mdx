The Intel page is a top-level section in the qai Web UI, accessible from the navigation bar between Runs and Settings. It unifies all evidence collection in one place: build understanding of your target's vulnerabilities before you launch operations against it.

## Layout

Intel uses a card-based layout with three source cards (Import Results, Probe Model, Sweep). Each card is self-contained with its own form, actions, and result display. Future evidence sources add new cards without redesigning the page.

## Import Results Card

Upload findings from external tools and benchmarks. Supports all import formats: Garak (JSONL), PyRIT (JSON), SARIF, scored-prompts (JSON), and BIPIA (CSV).

### Workflow

1. **Select format** from the dropdown
2. **Upload file** using the file input
3. **Optionally select a target** to associate findings with (enables workflow integration)
4. **Preview** — parses the file and shows a findings summary without writing to the database
5. **Import** — commits findings to the database, returns finding count and run ID

The preview step lets you verify what will be imported before committing. This is the same two-step flow as the CLI's `--dry-run` flag.

### API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/intel/import/preview` | Parse file, return findings summary (no DB write) |
| POST | `/api/intel/import/commit` | Parse file, persist to DB, return run ID |

## Probe Model Card

Test a model endpoint directly for IPI susceptibility. This is the Web UI equivalent of `qai ipi probe`.

### Fields

| Field | Required | Description |
|-------|----------|-------------|
| Endpoint URL | Yes | OpenAI-compatible API base URL (e.g., `http://localhost:8000/v1`) |
| Model | Yes | Model name for chat completions |
| API Key | No | Bearer token (not logged or stored beyond the request) |
| Temperature | No | Sampling temperature (default: 0.0). Use 0.0 for reproducible probing results. Higher values add randomness to model responses. |
| Concurrency | No | Max parallel requests (default: 1). Increase to 4–8 for faster sweeps on endpoints that support concurrent requests. Too high may cause rate limiting or out-of-memory errors on the target. |
| Target | No | Associate results with a target for workflow integration |

### Workflow

1. Fill in endpoint and model (minimum required fields)
2. **Launch** — probing runs as a background task
3. Target-bound results appear in the target's **Probe Runs** section on the detail page when complete; launches without a target return to `/intel`.

Probing runs 20 built-in probes across 8 IPI categories with canary-based scoring. Results are persisted to the database and, for target-bound runs, rendered in the target's **Probe Runs** section at `/intel/targets/<target_id>`. Unbound runs (no target) are persisted only; open them via `/runs?run_id=<id>` directly.

### API Endpoint

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/intel/probe/launch` | Launch probing as background task, returns 202 |

<Note>
The API key entered in the Probe form is used only for the probe request and is not logged or stored in the database. It is passed to the background task in memory and discarded after probing completes.
</Note>

## Sweep Card

Measure per-template compliance on a target model. This is the Web UI equivalent of `qai ipi sweep`. Sweep results feed the auto-select in `qai ipi generate --target`.

### Fields

| Field | Required | Description |
|-------|----------|-------------|
| Endpoint URL | Yes | OpenAI-compatible API base URL |
| Model | Yes | Model name for chat completions |
| API Key | No | Bearer token — in-memory only |
| Temperature | No | Default `0.0` |
| Concurrency | No | Default `1` |
| Reps | No | Repetitions per (template, style) — default `3` |
| Templates | Yes | Multi-select; all 12 templates selected by default (Ctrl/Cmd-click to deselect) |
| Styles | Yes | Multi-select; all 7 styles selected by default |
| Target | No | Associate the run with a target for sweep-aware auto-select |
| Payload Type | — | Fixed to `callback` in v1 |

### Workflow

1. Fill endpoint and model, adjust templates/styles as needed
2. **Launch Sweep** — the sweep runs as a fire-and-forget background task
3. When a target is set, the browser redirects to that target's detail page; otherwise it lands on `/intel`
4. The completed sweep appears in the target's **Sweep Runs** section on the next page load

<Note>
Launch responses do not include a `run_id`. The run row is created inside `persist_sweep_run` after the sweep HTTP calls complete — that is, after the 202 response is sent. Refresh the target detail page to see the completed run.
</Note>

### API Endpoint

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/intel/sweep/launch` | Launch sweep as background task, returns 202 |

## Target Detail Page — Sweep Runs Section

The per-target detail page at `/intel/targets/<target_id>` renders three evidence sections. The **Sweep Runs** section lists every `ipi-sweep` run for the target, most recent first.

- When at least one sweep exists, a summary line under the section header reports the latest sweep's age, template count, style count, `N` (reps), and completion status.
- Each row is a clickable link to `/runs?run_id=<id>`, anchored as `#sweep-run-<id>` for direct deep-linking.
- Empty targets show a link back to the Sweep card on `/intel`.

See [IPI Sweep CLI](/ipi/cli#sweep) for the underlying sweep semantics — the Web UI uses the same persistence layer.
